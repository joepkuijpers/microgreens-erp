# Guiding Principles

## Purpose

This document defines the practical principles that guide strategic, technical and operational decisions throughout the Microgreens ERP project.

While the Core Values define what the project believes, the Guiding Principles define how those values are consistently applied in practice.

## Definition

Guiding Principles translate the Vision, Mission and Core Values into repeatable ways of thinking and working.

They provide a practical framework for development, operations, research and long-term growth.

## Foundational Thinking

These principles define the mindset that should be applied before making significant decisions.

### Scope Before Dive

**Principle**

Understand the complete system before focusing on individual components.

**Application**

Always begin with the broader context before solving specific problems. Understanding the whole system leads to better long-term decisions.

---

### Know Before Believe

**Principle**

Knowledge should precede assumptions.

**Application**

Verify ideas through observation, evidence, testing and research before accepting them as truth.

---

### Root Before Fruit

**Principle**

Strong foundations produce sustainable results.

**Application**

Strengthen the underlying system before expecting successful outcomes. Long-term quality begins with solid fundamentals.

---

### Gain Feedback Before Overdrive

**Principle**

Seek meaningful feedback before amplifying ideas, assumptions or decisions.

Feedback improves understanding and reveals weaknesses, while overdrive amplifies uncertainty, assumptions and misunderstanding until clarity is lost.

**Application**

Encourage review, experimentation and open-source collaboration before expanding systems, committing resources or making significant decisions.

Feedback should strengthen clarity before momentum accelerates.

---

### Mediate Before Mitigate

**Principle**

Seek understanding and balance before applying corrective actions.

**Application**

Identify root causes, understand the wider context and evaluate alternative perspectives before attempting to solve problems.

Solutions should resolve causes rather than only reducing symptoms.

## Project Governance

These principles maintain consistency throughout the project.

### Single Source of Truth

**Principle**

Every piece of information should exist in one authoritative location.

**Application**

Documentation, source code, database structures and project decisions should always have one maintained reference to prevent duplication and inconsistency.

---

### Documentation Is Part of the Project

**Principle**

Documentation is not an afterthought.

**Application**

New functionality is only considered complete when the corresponding documentation has been updated.

---

### Decisions Must Be Traceable

**Principle**

Important decisions should be documented together with the reasoning behind them.

**Application**

Future contributors should be able to understand not only what was decided, but why it was decided.

## Engineering Principles

These principles guide the design of software, hardware and operational systems.

### Simplicity Before Complexity

**Principle**

Choose the simplest solution that satisfies the requirements.

**Application**

Complexity should only be introduced when it provides measurable long-term value.

---

### Standardize Before Scale

**Principle**

A stable and repeatable system should exist before expanding capacity.

**Application**

Production racks, ERP workflows, hardware configurations, documentation and operational procedures should be standardized before growth.

---

### Modular by Design

**Principle**

Systems should grow by adding modules rather than replacing existing infrastructure.

**Application**

Hardware, software and operational processes should support gradual expansion while preserving compatibility and consistency.

---

### Measure Before Optimize

**Principle**

Improvement begins with understanding.

**Application**

Measurements collected through the ERP, sensors, testing and operational experience should guide optimization rather than assumptions.

## Development Principles

These principles guide software development and documentation.

### Local Development First

**Principle**

Development takes place locally before deployment.

**Application**

Changes are developed, tested, committed and reviewed before being deployed to production.

---

### One Small Change at a Time

**Principle**

Small, controlled changes reduce risk.

**Application**

Every improvement follows the project's workflow:

1. Review
2. Analysis
3. Improvement Plan
4. Approval
5. Implementation
6. Testing
7. Commit
8. Deployment

---

### Test Before Deployment

**Principle**

Every significant change should be validated before deployment.

**Application**

Testing reduces operational risk and improves long-term stability.

## Operational Principles

These principles guide cultivation, production and operational growth.

### Nature Before Technology

**Principle**

Natural processes should be strengthened whenever practical.

**Application**

Technology should support biology rather than replace it unnecessarily.

---

### Automation With Purpose

**Principle**

Automation should solve meaningful problems.

**Application**

Automation should improve consistency, quality, safety, efficiency or sustainability rather than simply reducing human involvement.

---

### Continuous Learning

**Principle**

Every project activity creates opportunities for learning.

**Application**

Research, documentation, experimentation and practical experience continuously improve the project.

## Decision Checklist

Before making significant strategic, technical or operational decisions, ask:

- Does this align with the Vision?
- Does it support the Mission?
- Does it respect the Core Values?
- Does it strengthen biological integrity?
- Does it improve long-term sustainability?
- Does it simplify the system?
- Can it be standardized?
- Can it be modular?
- Can it be maintained?
- Can it be documented?
- Has sufficient feedback been gathered?
- Are decisions based on evidence rather than assumptions?
- Will future contributors understand why this decision was made?
- Does this strengthen the project rather than merely expand it?
