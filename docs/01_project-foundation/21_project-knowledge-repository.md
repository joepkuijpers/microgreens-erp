# Project Knowledge Repository

## Purpose

The Project Knowledge Repository serves as the central location for recording, preserving, and continuously improving the knowledge generated throughout the Microgreens ERP project.

Unlike the Project Foundation, which defines the philosophy and long-term direction of the project, this repository documents the current state of understanding.

Knowledge is expected to evolve.

New discoveries may confirm, expand, refine, or replace existing knowledge.

The objective is not to preserve conclusions.

The objective is to preserve learning.

## Philosophy

Knowledge is one of the project's most valuable resources.

Every observation, experiment, success, failure, and improvement contributes to a deeper understanding of the food chain.

The repository exists to ensure that knowledge is never unnecessarily lost and that future decisions can benefit from previous experience.

Knowledge should accumulate over time.

Each cultivation cycle should leave the project more informed than the previous one.

## Knowledge Development

Knowledge develops through continuous learning.

The project follows the progression below.

Reality
    ↓
Observation
    ↓
Measurement
    ↓
Data
    ↓
Information
    ↓
Knowledge
    ↓
Understanding
    ↓
Decision
    ↓
Action
    ↓
Improved Reality

Every stage contributes to the next.

Without observation there is no measurement.

Without measurement there is no reliable data.

Without data there is no information.

Without information there is no knowledge.

Without knowledge there can be no meaningful improvement.

## Knowledge Classification

Every topic within this repository should indicate its current level of confidence.

| Status | Description |
|---------|-------------|
| Verified | Supported through repeated observations, experiments, or reliable scientific literature. |
| Under Investigation | Initial evidence exists but additional validation is required. |
| Hypothesis | A proposed explanation that has not yet been verified. |
| Planned | Research has been identified but has not yet started. |

Knowledge should continuously move towards higher confidence as the project evolves.

## Repository Structure

Knowledge should be organised into clearly defined domains.

Examples include:

- Seed Knowledge
- Cultivation Knowledge
- Environmental Knowledge
- Sensor Knowledge
- Harvest Knowledge
- Freeze Drying Knowledge
- Packaging Knowledge
- Distribution Knowledge
- Energy Knowledge
- Research Knowledge
- Database Knowledge
- ERP Knowledge

Additional domains should be introduced whenever they improve the organisation of knowledge.

## Knowledge Entry Structure

Every knowledge entry should follow a consistent structure whenever applicable.

| Field | Description |
|-------|-------------|
| Topic | Subject being documented. |
| Status | Current confidence level. |
| Current Understanding | Summary of what is currently known. |
| Supporting Evidence | Observations, experiments, or literature supporting the knowledge. |
| Source | Reference to scientific literature, ERP data, or experiments. |
| Related Experiments | Experiments connected to the topic. |
| Open Questions | Questions that remain unanswered. |
| Planned Research | Future work intended to improve understanding. |
| ERP Implementation | How the knowledge is or will be integrated into the ERP. |
| Last Reviewed | Date of the latest review. |

Not every field is required for every topic.

The repository should remain practical rather than bureaucratic.

## Scientific Integrity

Knowledge should be based on observation whenever possible.

Assumptions should always be identified as assumptions.

Hypotheses should never be presented as facts.

Contradictory evidence should be preserved rather than removed.

The objective is continuous learning rather than confirming existing beliefs.

## Relationship with the ERP

The ERP is the primary source of operational knowledge.

Measurements, observations, production data, experiments, and historical records contribute to the knowledge contained within this repository.

The repository explains what has been learned.

The ERP records how that knowledge was obtained.

Together they create a continuously improving system.

## Continuous Improvement

The repository is a living document.

Knowledge should be reviewed, expanded, corrected, and refined as new evidence becomes available.

Its value increases with every cultivation cycle, experiment, observation, and improvement.

The purpose of the repository is not to become complete.

Its purpose is to continue growing.

## Relationship to Other Project Foundation Documents

The Project Foundation defines why the project exists and the principles that guide its long-term direction.

The Project Knowledge Repository documents what the project has learned.

Together they support evidence-based decision-making, scientific thinking, and continuous improvement throughout the Microgreens ERP project.
