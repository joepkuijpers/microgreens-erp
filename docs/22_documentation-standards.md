# Documentation Standards

## Purpose

This document defines the standards that govern the creation, maintenance, review, and evolution of documentation throughout the Microgreens ERP project.

The objective is to ensure that every document contributes to one coherent knowledge system rather than becoming an isolated source of information.

Documentation quality should remain independent of individual writing style.

Consistency is regarded as an engineering requirement rather than a stylistic preference.

---

# Philosophy

Documentation is part of the project.

Documentation is developed together with software, hardware, research, and operational procedures.

Documentation should preserve understanding.

Understanding should remain accessible to future contributors.

The purpose of documentation is not simply to describe what exists.

Its purpose is to explain why it exists, how it works, how it relates to other project components, and how it may responsibly evolve.

---

# Objectives

Documentation should:

- preserve knowledge;
- improve understanding;
- reduce ambiguity;
- support traceability;
- simplify maintenance;
- strengthen collaboration;
- support future automation;
- remain understandable across generations.

---

# Core Principles

Every document should have one primary responsibility.

Every chapter should answer one central question.

Every paragraph should strengthen its chapter.

Every concept should have one authoritative definition.

Every significant object should have one authoritative identifier.

Every statement should contribute to understanding.

# Writing Principles

Prefer principles over implementation.

Prefer timeless explanations over temporary instructions.

Prefer explanation over repetition.

Prefer references over duplication.

Prefer clarity over cleverness.

Prefer consistency over personal style.

Avoid unnecessary complexity.

Avoid marketing language.

Avoid assumptions.

Avoid undocumented abbreviations.

Use British English.

Write as if the document will still be read twenty years from now.

# Document Creation Policy

Creating a new document should be regarded as the final option rather than the default solution.

Before introducing a new document, the following questions should be considered:

- Does this concept already belong to an existing document?
- Does it answer a different fundamental question?
- Does a new document improve understanding?
- Does it reduce complexity?
- Does it reduce duplication?
- Does it strengthen the overall architecture?

A new document should only be created when it has one clear responsibility that cannot logically be incorporated into an existing document without weakening either document.

The objective is not to increase the number of documents.

The objective is to improve the quality, coherence, and maintainability of the project's knowledge architecture.
